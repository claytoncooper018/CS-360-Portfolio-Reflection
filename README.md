# CS-360-Portfolio-Reflection

# Inventory App — CS 360 Portfolio Artifact

The finished mobile application created for CS 360 is included in this repository and was turned in as an artifact for my Module Eight Journal portfolio. The full Android Studio project, containing the finished, functioning code (written in Project Three) and the finalized UI design (originally generated in Project Two), is contained in the ZIP file included here (`Clayton_Cooper_InventoryApp_Project3.zip`).

## Reflection

**App requirements and goals**
This app is a mobile inventory management application designed for use by individuals or small businesses. A straightforward, safe method of tracking products, categories, and stock levels via a phone without the requirement for an internet connection was the primary user demand. In order to fulfill this demand, the app allows users to create and log in, see and manage inventory on a dashboard, add new goods, and receive an automated low-stock SMS alert that lets them know when an item runs out even if they aren't using the app.

**Screens, features, and user-centered design**
A login/create account panel, an inventory dashboard, an add item screen, and a one-time SMS authorization screen comprise the app's three primary screens. Each user's inventory is kept private and connected to their own account via the login screen. Instead of requiring the user to open a separate screen for each minor modification, the dashboard offers a straightforward list/grid style with inline amount editing and delete actions because it was created with the most frequent task—scanning objects and modifying quantities—in mind. In order to prevent users from being sidetracked by irrelevant controls when adding a new item, the Add Item screen is a concentrated, one-purpose form. In order to respect user choice and adhere to Android best practices for runtime permissions, the SMS permission request is postponed until the first login (rather than being included into the installation). If permission is denied, the program still operates as intended. These decisions worked well because they made the most common actions—viewing and updating stock—quick and simple, while making less common actions—adding things and creating accounts—clearly distinct.

**Coding approach**
I constructed the application in layers: the model (`InventoryItem`), the data layer (a `DatabaseHelper` class wrapping SQLite, with specific CRUD methods for both the `users` and `inventory` tables), and the user interface (UI) layer (activities and an adapter for the inventory list). The activities just needed to use basic methods like `addItem()`, `updateItemQuantity()`, or `deleteItem()` rather than writing SQL directly because the database logic was contained in a single helper class. This kept the UI code clear and made it simpler to find issues. Additionally, instead of re-prompting on each login, I used SharedPreferences to monitor one-time status (i.e., if SMS authorization has already been sought). I'll keep using this separation-of-concerns strategy in my upcoming projects since it makes it simpler to test, debug, and expand an application by separating data access, business logic, and user interface code in separate classes.

**Testing process**
I used an Android emulator to test the app by manually going through each user flow, including creating an account, logging in with the right and wrong credentials, adding items, editing quantities up and down (including down to zero to confirm the SMS alert fires), deleting items, and denying SMS permission to make sure the app continued to operate normally afterward. Additionally, I evaluated edge circumstances such as creating an account with an already-existing username and submitting blank data. This procedure was important since it revealed genuine problems that would not have been apparent from reading the code alone, such as verifying that refusing a permission didn't cause the application to crash or prevent essential features.

**Where I had to innovate**
The low-stock SMS notice presented the most difficult design obstacle. The app couldn't rely on the availability of SEND_SMS permission because Android requires it to be asked at runtime and a user can refuse it at any time. Instead of making the SMS step dependent on the core update logic, I had to construct the quantity-update logic such that it was only one optional side effect of updating the database, wrapped in its own permission check. In this manner, the program never crashes or becomes stuck waiting for a permission that wasn't granted, and the inventory feature functions properly whether or not the user authorizes SMS authorization.

**Where I demonstrated strength**
The database layer (`DatabaseHelper`) is what I'm most happy with. It exposes a simple CRUD interface to the rest of the application, using parameterized queries to prevent SQL injection, and neatly separates user-account data from inventory data. The entire application relies on that class, and creating it with precise method names and documentation made it simpler to properly construct every other screen.
